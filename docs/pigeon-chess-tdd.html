Stack Decisions

Every decision below was evaluated against three constraints set in the GDD: solo/tiny team, web-first with Steam wrapper, and modifier logic must be server-authoritative. The decisions marked LOCKED are final. OPEN items are documented with trade-offs and must be resolved before Phase 0 begins.

Final Decisions — LOCKED

Frontend Framework React 18 + TypeScript 5 Component model maps well to game UI (board, modifier cards, draft UI). Large ecosystem. Tauri supports React natively. Strict TypeScript enforces game state typing from day one. LOCKED
Game Renderer Phaser 3 Phaser's Scene system is ideal for board rendering, pigeon animations, tile mutation effects, and fog overlays. Has built-in tweens, particles, and camera controls. Open Decisions — Must Resolve Before Phase 0

⚠ Open Decision 1 — Chess Engine: chess.js vs. Custom chess.js is battle-tested, handles all FIDE rules, and ships in days. But it has no concept of modifier hooks — every modifier that breaks standard rules requires monkey-patching chess.js internals or wrapping it in adapter logic that grows increasingly brittle.

Custom engine is designed from day one around the modifier state machine. It takes 2–4 weeks to build to parity with chess.js, but every modifier integrates cleanly.

Recommendation: Use chess.js for Phase 0 prototype only. Build the custom engine in parallel during Phase 1. The custom engine replaces chess.js before MVP launch. Do not ship chess.js to production — the modifier interaction debt will compound fast.
⚠ Open Decision 2 — Anti-Cheat Strategy All move validation is server-authoritative (non-negotiable). The open question is the depth of anti-cheat beyond that: engine analysis flagging (like Stockfish comparison), behavioral analysis (move timing patterns), or report-and-review only at launch.

Recommendation: Report-and-review at MVP. Stockfish integration in Ranked Season 2 when there's data to set meaningful thresholds. Modifier chaos makes engine comparison unreliable until baseline data is established.

ℹ Architecture Principle The Chess Engine and Modifier State Machine are pure TypeScript modules with no framework dependencies. Chess Engine

The custom chess engine lives in packages/game-core/engine/. It is the only source of truth for move legality. The server calls it. The client renders whatever the server emits. No exceptions.

Core Data Types

TypeScript — Core Types (packages/game-core/types/index.ts) 
type Square = `${'a'|'b'|'c'|'d'|'e'|'f'|'g'|'h'}${'1'|'2'|'3'|'4'|'5'|'6'|'7'|'8'}`;

Move Validation Pipeline

Every move passes through this pipeline on the server. If any stage rejects, the move is refused and the client is notified. No move reaches the board without completing all stages.

Move Validation Pipeline (pseudo-code) 
validateMove(state: GameState, move: Move): ValidationResult {

  // Stage 1 — Structural validity
  if (state.turn !== move.playerColor) return REJECT('not your turn')
  if (state.status !== 'active')      return REJECT('game not active')
  if (!state.pieces.has(move.from))    return REJECT('no piece at source')

  // Stage 2 — Piece ownership
  const piece = state.pieces.get(move.from)
  if (piece.color !== move.playerColor) return REJECT('not your piece')

  // Stage 3 — Pre-move modifier hooks
  // e.g. Opposite Day inverts direction, Heavy Crown requires 2-turn commit
  const preResult = runHooks('PRE_MOVE_VALIDATE', state, move)
  if (preResult.blocked) return REJECT(preResult.reason)

  // Stage 4 — Piece movement rules (engine or modifier override)
  const legalSquares = getLegalMoves(state, piece)
  if (!legalSquares.includes(move.to)) return REJECT('illegal move')

  // Stage 5 — Tile effect checks
  // e.g. cannot move INTO lava if not intended, swamp movement limits
  const tileResult = checkTileEffects(state, move)
  if (tileResult.blocked) return REJECT(tileResult.reason)

  // Stage 6 — Self-check guard (ALWAYS runs, cannot be bypassed by modifiers)
  const projected = applyMoveToState(state, move)
  if (isInCheck(projected, move.playerColor)) return REJECT('leaves king in check')

  return ACCEPT
}

King Safety — Absolute Rules

⛔ Non-Negotiable Engine Rules These are hard-coded engine constants that no modifier can override. They are enforced at Stage 6 of validation and cannot be hooked by any modifier in the state machine:

1. A move that leaves or places the real King in check is always illegal.
2. The real King (isRealKing: true) cannot be targeted by Betrayal, Rook Cannon, or The Snap.
3. Checkmate is evaluated only against the real King — Royal Guards (MOD-E003) are exempt.
4. The Sus modifier (MOD-D005) cannot disguise the real King.
5. Drop-to-check (MOD-E008 Moneyball) is validated at Stage 6 before the drop is accepted.

Modifier State Machine

This is the most critical architectural system in the game. Every modifier is a self-contained object that registers hooks into the game loop. New modifiers are added by creating a new file and registering hooks — no changes to engine core code required.

Modifier Interface

TypeScript — Modifier Definition Interface 
interface ModifierDefinition {
  id:          ModifierID;         // e.g. 'MOD-A001'
  name:        string;
  category:    'board' | 'curse' | 'ability' | 'fog' | 'army';
  pointCost:   number;             // negative = grants points (curses)
  curseRating: 0 | 1 | 2 | 3;

  // Called when modifier is added to a game — sets up initial state
  onActivate:  (state: GameState) => GameState;

  // Called at the START of every turn before the player moves
  onTurnStart?: (state: GameState, color: Color) => GameState;

  // Called BEFORE a move is validated — can block or transform the move
  onPreMoveValidate?: (state: GameState, move: Move) => PreMoveResult;

  // Called AFTER a legal move is confirmed but BEFORE board state updates
  onPreMoveApply?: (state: GameState, move: Move) => GameState;

  // Called AFTER board state is updated with the move result
  onPostMove?: (state: GameState, move: Move) => GameState;

  // Called AFTER a capture occurs (either player capturing)
  onCapture?: (state: GameState, captured: Piece, by: Piece) => GameState;

  // Called when a pawn reaches the promotion rank
  onPromotion?: (state: GameState, pawn: Piece, toSquare: Square) => GameState;

  // Called at the END of every turn after all effects resolve
  onTurnEnd?: (state: GameState, color: Color) => GameState;

  // Called when the game ends — cleanup, bonus Feathers, etc.
  onMatchEnd?: (state: GameState, result: MatchResult) => MatchEnd;

  // Fog modifiers: which squares are visible to a given color?
  getVisibleSquares?: (state: GameState, color: Color) => Set<Square>;
}

Example Modifier Implementation

Example — MOD-B002 "Distracted Piece" (Gerald) 
const DistractedPiece: ModifierDefinition = {
  id:          'MOD-B002',
  name:        'Distracted Piece',
  category:    'curse',
  pointCost:   -2,
  curseRating: 2,

  onActivate: (state) => {
    // Store Gerald's state in modifierState
    state.modifierState['MOD-B002'] = {
      geraldTarget: null,
      geraldLandCount: 0   // for Gerald's leaderboard
    };
    return state;
  },

  onTurnStart: (state, color) => {
    // Gerald only bothers the CURSED player
    const cursedColor = state.modifierState['MOD-B002'].cursedColor;
    if (color !== cursedColor) return state;

    // Pick a random non-ghost, non-pigeon piece to land on
    const eligible = [...state.pieces.values()].filter(p =>
      p.color === cursedColor && !p.isGhost &&
      p.type !== 'pigeon'  // MOD-B002 × MOD-E001 interaction
    );

    const target = randomFrom(eligible);
    state.modifierState['MOD-B002'].geraldTarget = target.id;
    state.modifierState['MOD-B002'].geraldLandCount++;

    // Emit visual event to client
    state.queueClientEvent({ type: 'GERALD_LANDS', pieceId: target.id });
    return state;
  },

  onPreMoveValidate: (state, move) => {
    const ms = state.modifierState['MOD-B002'];
    const piece = state.pieces.get(move.from);

    // If the targeted piece tries to move, block it
    if (piece?.id === ms.geraldTarget) {
      return { blocked: true, reason: 'Gerald is sitting on this piece',
               clientEvent: { type: 'GERALD_BLOCKS' } };
    }

    // Special: player clicks Gerald to shoo him (spends turn)
    if (move.type === 'SHOO_GERALD') {
      ms.geraldTarget = null;
      state.queueClientEvent({ type: 'GERALD_LEAVES' });
      return { blocked: false, consumesTurn: true };
    }

    return { blocked: false };
  }
};

The 9 Hook Types — Usage by Modifier

onActivate
Runs once when modifier is added. Sets up initial state in modifierState record.
Used by: ALL modifiers
onTurnStart
Runs at the start of each turn before player input. Ticks, landings, spread effects.
Used by: Cordyceptic Spread, Gerald, Storm, Floor Is Lava rotation, YOU DIED autonomous move
onPreMoveValidate
Can block or transform a move before engine validation. First line of modifier interference.
Used by: No I Don't Think I Will, Heavy Crown, Conscientious Objector, Swamp range, Frozen Tundra
onPreMoveApply
State mutations before board updates. Teleports, mirroring, etc.
Used by: Vent Tiles, Bandersnatch Protocol, Gravitational Anomaly
onCapture
Fires when any piece is captured. Economy effects, chain capture checks.
Used by: BOY. (chain check), Moneyball (token gen), Smoke Screen (cloud spawn), They Were The Traitor (timer)
onPromotion
Fires when a pawn reaches the back rank. Promotion rule enforcement.
Used by: Reject Queen Embrace Knight, Conscientious Objector carry-through
onPostMove
After all board updates. Chain continuations, secondary effects, double-turns.
Used by: BOY. chain, Twin Telepathy bishop mirror, This Is Necessary double-turn
onTurnEnd
Cleanup, cooldown tick, duration expiry, zone checks.
Used by: ALL ability cooldowns, No-Fly Zone duration, Frozen thaw, Floor Is Lava rotation counter
getVisibleSquares
Fog modifiers override this to return a custom visibility set per color.
Used by: It's Dark And I'm Scared, Cat Mode, Known Unknowns, Big Beak Is Watching

Real-Time Layer

All real-time communication uses Socket.IO rooms. Each active game is a room. Players join their room on match start. Spectators (post-launch) join as read-only listeners.

Fog Filtering — Per-Recipient State

ℹ Critical: Never Send Full State to Both Players When any Fog modifier is active, the server runs getVisibleSquares(state, color) for each player and strips hidden piece positions before emitting. A player must never receive the position of a piece they shouldn't be able to see — even as dead data in a JSON payload. This is server-enforced, not client-enforced.
Fog State Filtering (server/game/emitState.ts) 
function emitGameState(room: GameRoom, state: GameState) {
  for (const [playerId, color] of room.players) {

    const visibleSquares = computeVisibleSquares(state, color);

    const filteredState: ClientGameState = {
      ...state,
      pieces: filterPieces(state.pieces, visibleSquares, color),
      tiles:  filterTiles(state.tiles, visibleSquares),
      // Ghost pieces: opponent sees ghost as real; owner sees it as ghost
      // Sus (MOD-D005): opponent sees disguised type, owner sees true type
    };

    room.socket.to(playerId).emit('game:state', filteredState);
  }
}

Matchmaking Service

Matchmaking runs as a lightweight service loop in Redis. Players join a queue key, the loop pairs them by ELO range, and creates a game room. All matchmaking state lives in Redis — if the server restarts, queues drain and players re-queue automatically.

Matching Algorithm

Matchmaking Loop (pseudo-code, runs every 2s) 
for each mode in ['ranked_blitz', 'ranked_rapid', 'casual']:

  queue = redis.zrange(mode + ':queue', 0, -1, 'WITHSCORES')
  // Queue stored as sorted set: score = ELO rating

  for each player in queue:
    waitTime = now - player.queuedAt

    // Expand ELO range the longer they wait
    eloRange = 100 + (waitTime / 10s) * 25  // +25 ELO per 10 seconds
    eloRange = min(eloRange, 400)            // cap at ±400

    candidates = redis.zrangebyscore(
      mode + ':queue',
      player.elo - eloRange,
      player.elo + eloRange
    ).filter(c => c.id !== player.id)

    if candidates.length > 0:
      opponent = closest(candidates, player.elo)
      createGame(player, opponent, mode)
      removeFromQueue(player, opponent)

Replay System

Every match is recorded as a deterministic event stream. The replay system re-feeds the event stream through the game engine to reconstruct any board state at any point in the match. This serves four purposes: player replays, puzzle creation, anti-cheat review, and shareable highlight clips.

Event Stream Structure

Replay Event Record 
interface ReplayEvent {
  seq:       number;       // monotonic sequence number
  turn:      number;       // game turn when event occurred
  timestamp: number;       // Unix ms
  type:      EventType;
  payload:   unknown;      // typed per EventType
}

type EventType =
  | 'MOVE'              // { from, to, piece, captured? }
  | 'ABILITY_USED'      // { modifierId, pieceId, target? }
  | 'MODIFIER_TRIGGER'  // { modifierId, effect, affected[] }
  | 'PIECE_DEFECTED'    // MOD-B006 { pieceId, fromColor, toColor }
  | 'GERALD_LANDS'      // MOD-B002 { pieceId }
  | 'STORM_SHRINKS'     // MOD-A005 { removedSquares[] }
  | 'VENT_TRIGGERED'    // MOD-A003 { from, to, pieceId }
  | 'PROMOTION'         // { from, to, pawnId, promotedTo }
  | 'DRAW_OFFER'        // { byColor }
  | 'RESIGN'            // { byColor }
  | 'GAME_END';         // { result, reason }

Replay Determinism Rules

✓ Determinism Guarantee The game engine is a pure function: given an initial state and a sequence of events, it always produces the same final state. This means:

• All randomness (Gerald targeting, Cordyceptic spread, YOU DIED piece selection) uses a seeded PRNG — the seed is stored with the match and included in the event stream.
• No modifier reads from external state (time, network, etc.) — only from GameState and the event payload.
• Replay playback is byte-identical to the original match at every turn.

Roguelike Systems

Run State Machine

RUN_START
Run initialised. Node map generated. Starting modifier assigned. Feathers set to 0.
Stored in: roguelike_runs
NODE_SELECT
Player chooses next node from available branches. Node type determines next state.
Transitions to: MATCH | EVENT | SHOP | REST | BOSS
IN_MATCH
Standard match in progress. Run state paused until match resolves.
On win: NODE_COMPLETE · On loss: RUN_LOST
IN_EVENT
Flock Event node. Present 2–3 choices. Apply chosen consequence to run state.
Consequences: modifier add/remove, feather gain/loss, curse add
IN_SHOP
The Roost. Player spends Feathers to buy/sell modifiers and relics.
Items: from unlocked modifier pool + relics table
IN_REST
The Perch. Player chooses: remove one curse OR preview next opponent's loadout.
Single choice: cannot do both
NODE_COMPLETE
Feathers awarded. Madness meter updated. Next node options revealed.
Returns to: NODE_SELECT
TOWER_CLEAR
Boss defeated. Prestige Tokens awarded. Tower 5 = RUN_WON.
Unlocks: meta-progression rewards, Viktor story segment
RUN_LOST / RUN_WON
Run ends. Prestige Tokens awarded (win or loss). Run archived. Meta-progression updated.
Stored in: roguelike_runs (status updated)

Node Map Generation

Node Map Algorithm 
// Generates a Slay the Spire-style branching node map for one Tower
generateNodeMap(tower: number, seed: string): NodeMap {

  const ROWS = 8           // nodes per tower
  const WIDTH = 3          // max parallel paths
  const rng = seededRNG(seed)

  const distribution = {
    MATCH: 0.60,
    EVENT: 0.15,
    SHOP:  0.15,
    REST:  0.10
  }

  // Rules:
  // Row 1 is always MATCH (start fighting immediately)
  // Row 4 is always SHOP (mid-tower upgrade opportunity)
  // Row 8 is always BOSS
  // No two consecutive REST nodes
  // At least one SHOP in every tower
  // Each node connects to 1–3 nodes in the next row

  return buildGraph(ROWS, WIDTH, distribution, rng)
}

Madness Meter Logic

The Madness Meter is derived from the total curse rating of all active modifiers. It is not a stored value — it's computed on each state read.

Madness Calculation 
computeMadness(activeModifiers: ModifierInstance[]): number {
  const totalCurse = activeModifiers
    .reduce((sum, m) => sum + m.definition.curseRating, 0)

  // Max theoretical curse: all 8 curses active = curse ratings of 3+3+2+2+2+1+1+1 = 15
  // Scale to 0–100
  return Math.min(100, (totalCurse / 15) * 100)
}

// Madness thresholds
// 0–25:  Viktor is composed. Normal dialogue.
// 26–50: Viktor is unsettled. Board has minor visual glitches.
// 51–75: Viktor is unravelling. Dialogue becomes erratic. Board shows pigeon shadows.
// 76–99: Viktor has accepted it. Hidden modifiers available. Board is deeply wrong.
// 100:   Full Flock. Secret modifier triggers. Pigeons everywhere.

Auth & Accounts

Auth Flow

[ User ] → clicks "Sign In" → [ Clerk hosted UI ] │ OAuth provider (Google / Discord / Steam) │ Clerk issues JWT → stored in httpOnly cookie │ [ React app ] reads Clerk session → attaches JWT to all API calls │ [ Fastify ] verifies JWT via Clerk SDK on every protected route │ [ Fastify ] looks up user by clerk_id in PostgreSQL users table │ New user → creates users row + default elo_ratings rows

Socket Auth

Socket.IO connections authenticate during the handshake by passing the Clerk JWT as an auth parameter. The server middleware verifies the JWT before allowing the socket to join any game room. Unauthenticated sockets can observe public spectator rooms only (post-launch).

Guest Play

Guests (no account) can play VS Bots and Puzzle Mode with a temporary session. Casual and Ranked multiplayer require an account. Guest progress is not saved. A persistent upsell prompt appears after each guest match.
